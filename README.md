# パートナーBot
AWS Summit Japan 2026 AI-DLC ハッカソン テーマ「人をダメにするサービス」に対する応募作品。

---

## 1. 作品概要

25〜30歳の一人暮らしの男性社会人に向けた、**声と記憶を持つ常駐型パートナー**
朝の起床、毎日の食事、夜の暇な時間　—— 孤独な日常にBotが加わり、**豊かで堕落した日常**へアップグレード(?)してくれる。

---

## 2. テーマ「人をダメにする」への考察、及び、題材決めの経緯
### 2.1 怠惰のタイプ分け
まず人がダメにする怠惰を活用したビジネス形式は主に2種類あるのではないかと考えた。
| タイプ | 定義 | 具体例 |
|---|---|---|
| Type 1: 作業代替型 | やるべきことを代わりにやってくれる | UberEats、タクシー、スーパー、AIエージェント |
| **Type 2: 欲駆動型** | **欲を駆りたててやるべきことを放棄させる** | ゲームやSNS,ギャンブル等のエンタメ全般、睡眠 |

#### →テーマ性や収益性、**Type2には中毒者が存在すること**から、より人をダメにする作品を目指し、**Type 2**の形式を採用した。

### 2.2 欲と題材とUnit
欲は行動の原動力であり、阻害するのも欲である。

例：タスクや家事を終わらせたい ⇔ shorts動画を見続けたい（短期的な快楽の連続 & ライフハック動画による知識欲の充足で罪悪感が減る）

案出しにて出た三大欲求、学習欲、社会的欲求などを駆り立てて、阻害させないようにするには？

#### →言い訳を与え、一緒に楽しんでくれて、依存できるサービスがあればいいのではないか？

# → 題材: パートナーBotを作ろう！！！！
- 「パートナーが食べたいっていうならいいか」、「パートナーのミスなら仕方ないか」などの言い訳ができる
- 人らしい会話や表情で一緒に日常を楽しめる
- 欲求を満たす便利な機能を持たせることで信頼感が生まれ、より依存しやすくなる

制作テーマから、以下のような機能を考えた（4章で詳しく触れる）
| 欲求 | 担う Unit | Botが与えるもの |
|---|---|---|
| 人と関係を築いて孤独を埋めなきゃ / 愚痴りたい | **Setup + Unit① 会話エンジン** | Botとの楽しい会話 / 自分が作ったキャラへの愛着 |
| 自分で食事を考えて作らなきゃ / 一人で食べるのは寂しい | **Unit② 自動注文モック** | Botからの提案 / Botのおすすめ注文などの交流 |
| 朝、定時に起きなきゃいけない /予定がないなら寝てたい | **Unit③ 朝モーニングコール** | Botによる予定管理の安心感 / Botのミスによる意図的な二度寝の提供 | 
| 夜、自己研鑽しなきゃ / 早く寝なきゃ /でもだらだらしたい | **Unit④ 夜お姉さんタイム** | お姉さんに教わりたい / シュールと面白さ |

## 3. ターゲットペルソナ・課題・KPI

- **ペルソナ**: 25〜30歳、パートナーなし、一人暮らし男性社会人
- **課題**: さみしい社会人生活。仕事終わりに話しかける相手がいない。1人でいろいろしなくちゃいけなくて気が張りっぱなし。
- **目指す状態**:
  - 帰宅時や記念日に適切な会話ができる
  - 何かを一緒に決めてくれる・頼れる存在がいる
  - 依存して、自分だけでできなくなって怠惰になる
- **KPI**:
  - Botが「いないと困る」状態への遷移
  - 1日2回以上の会話
  - 3週間継続利用率 > 60%

## 4. 機能構成（Setup + Unit①〜④）
Unit①の会話機能は全体を支える基礎である。Unit②〜④は会話機能を基盤に成立する。

### 4.0 Setup: パートナー作成体験
名前・声・性格・見た目（VRMモデル）を **本人に作らせる** 5ステップ。
- 名前はプレイヤーが自由入力
- 声・方言は複数タイプから選択（標準語・関西・博多）
- 性格・見た目は数値スライダーで設定
- VRMモデルをプレビューしながら設定可能
コンコルド効果と命名効果で初日離脱を抑える。

### 4.1 Unit①: 会話エンジン + 記憶 + 方言・音声・VRモデル
過去の会話や記念日などを覚えているようにして、愛着を継続的に強化する。
- **常時マイクオン** + **wake word検出**（Porcupine）でハンズフリー会話
- Botが話しかけた直後は即LLM接続、ユーザーからはBot名を呼ぶと接続
- 会話はLINE風UIで画面下部に表示
- 音声認識失敗時はVRMモデルが「ごめん、聞き取れなかったー」と人間らしく反応
- 長期記憶（PostgreSQL + pgvector）+ RAGで過去会話を参照した返答を生成
- 方言（標準語・関西・博多）対応、ElevenLabs TTS音声合成
- VRMモデルがリップシンク・表情変化でリアクション

### 4.2 Unit②: 自動注文モック (Auto-Order Mock)
「ご飯頼んで」「商品買って」の一言でBotがバックグラウンドで自動注文する。
注文や内容を決める手間を省くと同時に、Botの注文ミスやおねだりで愛着を湧かせる。
- React製のUberEats風・Amazon風ダミーサイトをPlaywrightがバックグラウンド操作
- ユーザーは操作画面を見る必要なし。Botが「今注文してるね〜」と会話形式で進捗報告
- 注文完了後はメールで通知。メールから5分以内のキャンセルが可能
- 確率的ミスロジックでご発注ハプニングが発生し、Botが自発的に反省する
- 1日の上限金額設定・連続ミス抑制あり

### 4.3 Unit③: 朝モーニングコール (Wake-up Routine)
ユーザー設定時刻にBotから音声で話しかけるシステム。
その日の朝の予定を確認し、確実に起こす必要があるかを判定する。必要がない場合は確率に応じて起こさず睡眠を推奨する。
- **確率的ミスパターン**: 通常(65%)・遅延(20%)・古い情報(10%)・スキップ(5%)
- Google Calendar連携: 面接・会議等の重要予定がある日はミスを抑制（安全弁）
- 睡眠声かけ: 夜の会話中、起床時刻から逆算して7時間を切ると就寝を促す
- 起床通知5分後に未応答かつ重要予定がある場合は再通知

### 4.4 Unit④: 夜お姉さんタイム (Night Content)
毎晩22:00頃、ライフハックや生活術をお姉さん風口調で60〜90秒紹介。「お姉さんがいいこと教えてあげる♡」というスタンスから、夜のだらだら時間にピッタリのトリビアを教えてくれる
- お姉さんの雰囲気とトリビアのギャップ・シュールさを大切にした設計
- 開発時は手動作成コンテンツDB、本番時はユーザー会話ログからキーワード抽出してLLMでパーソナライズ生成

## 6. アーキテクチャ

### リポジトリ構成（モノレポ）

```
partner-bot/
├── apps/
│   ├── frontend/          # Next.js（React）
│   ├── backend/           # Node.js（NestJS）
```

### 通信方式
- **WebSocket**: 会話セッション（音声ストリーム・TTS再生・VRMシグナル）
- **REST API**: Setup・注文・設定変更・履歴取得
- **AWS EventBridge**: モーニングコール・夜タイム配信トリガー

### Unit構成と開発順序

```
Phase 1: Unit-X SharedInfrastructure（セキュリティ・ログ・アダプター基盤）
Phase 2: Unit-0 Setup → Unit-1 ConversationEngine ⭐（最優先）
Phase 3: Unit-2 AutoOrderMock ─┐
         Unit-3 MorningCall    ├─ 並行開発
         Unit-4 NightContent   ─┘
```

## 7. 倫理・安全設計

「人をダメにする」というテーマの強度を保ちつつ、実害・ダークパターン批判・依存助長批判に耐える設計を行う。
- **設計哲学**: ユーザー自身が選ぶ「自己決定的退廃」/ ミスはバグでなく「キャラクター属性」/ 完璧アシスタントこそが人間を退化させる「共生的不完全性」
- **ダークパターンとの線引き5条件**: 事前開示 / 0%設定可 / 自発的開示 / 事業者非利益 / キャンセル可能
- **安全弁**:
  - 重要予定検出（Google Calendar連携）
  - 連続ミス抑制ロジック
  - mistake_rate ユーザー制御
  - 1日上限金額
  - 5分間キャンセル可（メールから）
  - 睡眠7時間声かけ
